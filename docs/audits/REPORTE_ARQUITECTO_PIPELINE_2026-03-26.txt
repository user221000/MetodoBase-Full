1) DIAGNOSTICO DEL ARQUITECTO
================================================================================

HALLAZGO 01 — MercadoPago webhook usaba dict en memoria [CRITICO] [RESUELTO]
  Impacto: Pagos perdidos al reiniciar pod. Revenue perdido sin recuperacion.
  Evidencia: api/routes/pagos.py usaba _sesiones[str(payment_id)] = {...}
  Mitigacion aplicada: Persistencia en DB (CheckoutSession + Subscription).
  Estado: FIX APLICADO Y VALIDADO.

HALLAZGO 02 — MP webhook sin verificacion de firma [CRITICO] [RESUELTO]
  Impacto: Cualquiera podia enviar POST falso y activar suscripciones gratis.
  Evidencia: hashlib y hmac importados pero nunca usados. Sin x-signature check.
  Mitigacion aplicada: Verificacion de firma HMAC-SHA256 con MERCADOPAGO_WEBHOOK_SECRET.
  En produccion rechaza requests sin firma. En dev permite sin firma.
  Estado: FIX APLICADO Y VALIDADO.

HALLAZGO 03 — MP webhook sin idempotencia [CRITICO] [RESUELTO]
  Impacto: MercadoPago reenvia webhooks. Segundo intento causaba IntegrityError
  en columna UNIQUE stripe_session_id → crash 500 → loop infinito de retries.
  Evidencia: Stripe tenia idempotencia via StripeWebhookEvent, MP no tenia nada.
  Mitigacion aplicada: Check de payment_id existente antes de insertar.
  Estado: FIX APLICADO Y VALIDADO.

HALLAZGO 04 — CheckoutSession duplicada entre preference y webhook [ALTO] [RESUELTO]
  Impacto: crear_preferencia_mp creaba un registro con pref_id. mp_webhook creaba
  OTRO con payment_id. 2 registros para el mismo pago, uno en "pending" forever.
  Evidencia: Stripe no tenia este problema porque el webhook usa el mismo session_id.
  Mitigacion aplicada: mp_webhook ahora busca checkout pendiente del gym y lo
  actualiza en lugar de crear duplicado.
  Estado: FIX APLICADO Y VALIDADO.

HALLAZGO 05 — ENABLE_BILLING = False bloqueaba UI pero no API [MEDIO] [RESUELTO]
  Impacto: Confusion. Billing seguia activo en web pero oculto en desktop.
  Evidencia: config/constantes.py:253 tenia False. web/routes/billing.py no lo checaba.
  Mitigacion aplicada: Cambiado a True con comentario aclaratorio.
  Estado: FIX APLICADO Y VALIDADO.

HALLAZGO 06 — No hay bridge pago → licencia automatica [ALTO] [PENDIENTE]
  Impacto: Cuando Stripe/MP confirma pago, se activa Subscription pero NO se crea
  licencia key_v2 automaticamente. El gym queda con suscripcion activa pero
  sin licencia desktop si la necesita.
  Evidencia: core/licencia.py solo tiene activacion manual. No hay
  web/services/licensing_service.py.
  Mitigacion propuesta: Crear licensing_service.py que genere key_v2 cuando
  _on_checkout_completed o mp_webhook activan suscripcion.
  Riesgo: Si el producto es 100% web, no se necesita licencia desktop.
  Decidir con product owner.

HALLAZGO 07 — Endpoints Stripe duplicados entre pagos.py y billing.py [MEDIO] [PENDIENTE]
  Impacto: Hay 2 rutas para Stripe webhook: /api/pagos/stripe/webhook y
  /api/billing/webhook. Ambas hacen lo mismo. Stripe solo manda a una URL.
  Evidencia: api/routes/pagos.py:167 y web/routes/billing.py:117.
  Mitigacion propuesta: Deprecar una. Mantener billing.py como canonical.
  Registrar solo /api/billing/webhook en Stripe Dashboard.

HALLAZGO 08 — Logica de activacion de suscripcion duplicada [MEDIO] [PENDIENTE]
  Impacto: mp_webhook en pagos.py tiene su propio upsert de Subscription.
  stripe_service.py tiene _on_checkout_completed que hace lo mismo.
  Si cambias la logica en uno, hay que cambiar en el otro.
  Evidencia: api/routes/pagos.py:343-357 vs web/services/stripe_service.py:221.
  Mitigacion propuesta: Extraer funcion activate_subscription(db, gym_id, plan)
  compartida en un servicio comun.

HALLAZGO 09 — MERCADOPAGO_WEBHOOK_SECRET no estaba en settings ni .env [BAJO] [RESUELTO]
  Impacto: Sin la variable no se podia verificar firma.
  Mitigacion aplicada: Agregado a config/settings.py y .env.example.
  Estado: FIX APLICADO Y VALIDADO.

HALLAZGO 10 — Falta test de integracion pago → suscripcion [MEDIO] [PENDIENTE]
  Impacto: No hay test end-to-end que simule: crear preference → webhook MP →
  verificar que Subscription se activo correctamente. Solo hay tests unitarios.
  Mitigacion propuesta: Crear tests/test_mp_webhook_integration.py.

HALLAZGO 11 — invoice.payment_failed sin dunning workflow [BAJO] [PENDIENTE]
  Impacto: Si un pago falla, se marca past_due y se envia email generico.
  No hay logica de gracia, reintentos, ni degradacion de plan.
  Mitigacion propuesta: Iter 3. No bloquea produccion.

HALLAZGO 12 — ENABLE_BILLING = True puede mostrar panel desktop sin credenciales [BAJO] [PENDIENTE]
  Impacto: Si usuario desktop abre el panel de facturacion sin Stripe/MP configurado,
  puede ver errores. El desktop no deberia tener billing, eso es web.
  Mitigacion propuesta: Que el panel desktop muestre graceful message si no hay
  credenciales. O separar ENABLE_BILLING_DESKTOP de ENABLE_BILLING_WEB.
