1. RESUMEN EJECUTIVO
================================================================================

MetodoBase es una plataforma SaaS de nutrición para gimnasios construida con:
  - Backend: FastAPI + SQLAlchemy + Alembic
  - Frontend Web: Jinja2 + HTMX + CDN libs
  - Desktop: PySide6

SCORE DE PRODUCCIÓN: 5.5/10 — NO ESTÁ LISTO PARA PRODUCCIÓN

La aplicación tiene una base técnica sólida (1030 tests pasando, middleware
de seguridad implementado, RBAC funcional), pero presenta BLOQUEANTES CRÍTICOS
en: multi-tenancy, datos hardcodeados del desarrollador, configuración de pagos
incompleta, y migración de base de datos sin finalizar.

2. HALLAZGOS CRÍTICOS (Bloquean deploy)
================================================================================

C-01 | BRANDING HARDCODEADO DEL DESARROLLADOR
     Archivo: config/branding.json
     Impacto: TODOS los gyms clientes verían "FitnessGym Real del Valle",
              el teléfono personal (5217441614117), el email personal
              (oscar_autumn@outlook.com), y la dirección física del
              desarrollador.
     Detalle: El archivo branding.json contiene datos estáticos:
              - nombre_gym: "FitnessGym Real del Valle"
              - telefono: "5217441614117"
              - email: "oscar_autumn@outlook.com"
              - direccion: "C. Valle De San José 1329B..."
              - cuota_mensual: 800
     Fix: Mover branding a tabla en base de datos por gym_id.
          Cada gym debe tener su propio registro de branding cargado
          dinámicamente al login.

C-02 | EMAIL PERSONAL EN TEMPLATES HTML (VISIBLE A CLIENTES)
     Archivos:
       - web/pages/landing.html (línea 219): "Oscar_Autumn@outlook.com"
       - web/templates/suscripciones.html (líneas 203, 208):
         WhatsApp hardcodeado "5217441614117"
     Impacto: Los usuarios finales ven datos personales del desarrollador,
              no del gym que contrató el servicio.
     Fix: Cargar datos de contacto desde branding dinámico (DB) y usar
          variables Jinja2: {{ gym.contacto.email }}, {{ gym.contacto.whatsapp }}

C-03 | STRIPE PRICE IDs SON PLACEHOLDERS (PAGOS NO FUNCIONAN)
     Archivo: config/constantes.py (líneas 103-127)
     Detalle: Los IDs de precio de Stripe son strings inventados:
              - "price_starter_monthly_29"
              - "price_profesional_monthly_59"
              - "price_clinica_monthly_129"
              Estos NO son IDs reales de Stripe y los webhooks fallarán.
     Fix: 1) Crear productos/precios en Stripe Dashboard
          2) Mover IDs a variables de entorno:
             STRIPE_PRICE_STARTER, STRIPE_PRICE_PRO, STRIPE_PRICE_CLINICA
          3) Validar que existan al arrancar el servidor

C-04 | MERCADOPAGO INCOMPLETO — NO CREA LICENCIAS
     Archivo: api/routes/pagos.py (línea 262)
     Contenido: "# TODO: Verificar pago con SDK y auto-crear licencia."
     Impacto: Los pagos por MercadoPago se procesan pero NO generan la
              licencia automáticamente. El cliente paga y no recibe nada.
     Fix: Implementar webhook de MercadoPago con verificación y
          auto-creación de licencia (como ya existe para Stripe).

C-05 | SECRET_KEY CON DEFAULT INSEGURO
     Archivo: config/settings.py (líneas 27-38)
     Detalle: En desarrollo usa "metodobase_dev_secret_2026_localhost_only"
              y el SALT usa "METODO_BASE_2026_CH".
              Si por error no se configuran las env vars en producción,
              el sistema arranca con secretos predecibles.
     Fix: En producción, si WEB_SECRET_KEY o METODO_BASE_SALT no están
          configurados, el servidor DEBE crashear inmediatamente.
          Verificar que _require_in_prod() realmente lanza excepción.

C-06 | LICENSE SALT HARDCODEADO EN CÓDIGO
     Archivo: core/licencia.py (~línea 30)
     Detalle: SALT_MASTER tiene fallback hardcodeado "METODO_BASE_2026_CH"
              Si env var no existe, todas las licencias usan el mismo salt,
              permitiendo forjar licencias válidas.
     Fix: Eliminar el default. Requerir METODO_BASE_SALT como obligatorio.

C-07 | DEMO USERS CON CREDENCIALES DÉBILES
     Archivo: web/auth.py (líneas 169-182)
     Detalle: Función _seed_demo_users() crea:
              - gym@test.com / test123
              - usuario@test.com / test123
              Se activa con METODOBASE_SEED_DEMO=1, pero NO valida que
              el entorno sea desarrollo.
     Fix: 1) Agregar check: if settings.is_production: return
          2) Mejor: eliminar la función completamente
          3) Crear script separado para seeds de desarrollo

C-08 | DATABASE_URL CAE A SQLITE EN PRODUCCIÓN
     Impacto: Si DATABASE_URL no está configurado o tiene un typo,
              el sistema silenciosamente usa SQLite local. En Railway
              con múltiples pods, cada pod tendría su propia DB vacía.
     Fix: En producción, requerir DATABASE_URL con postgresql://
          Validar el scheme al arrancar.

C-09 | ALEMBIC MIGRATIONS NO SE EJECUTAN EN DEPLOY
     Archivos: Dockerfile, Procfile, railway.toml
     Detalle: El CMD del Dockerfile es: python web/main_web.py --no-browser
              No hay paso de "alembic upgrade head" antes de arrancar.
              Cambios de schema requieren intervención manual.
     Fix: Agregar release command en railway.toml:
              [deploy]
              releaseCommand = "alembic upgrade head"
          O crear un entrypoint.sh que ejecute migraciones antes de iniciar.

C-10 | create_all() BYPASSA EL SISTEMA DE MIGRACIONES
     Detalle: El código llama Base.metadata.create_all() directamente,
              creando tablas fuera del control de Alembic. Esto causa
              que Alembic piense que las migraciones no se han aplicado.
     Fix: Eliminar create_all() en producción. Solo Alembic debe manejar
          el schema.

C-11 | HEALTH CHECK EXPONE ERRORES INTERNOS
     Archivo: api/app.py (líneas 145-165)
     Detalle: El endpoint /health devuelve str(e) cuando hay errores de DB,
              exponiendo información interna del sistema (paths, queries, etc.)
     Fix: Retornar solo {"status": "error"} sin detalles. Loguear
          internamente el error completo.

3. HALLAZGOS ALTOS (Requieren fix pre-producción)
================================================================================

H-01 | JWT TOKEN TYPE NO VALIDADO ESTRICTAMENTE
     Archivo: web/auth.py (~línea 390)
     Detalle: verificar_token() usa payload.get("type", "access") — un token
              sin campo "type" es aceptado como access token.
     Fix: Requerir campo "type" explícito. Si no existe: return None.

H-02 | LICENCIAS ALMACENADAS EN MEMORIA (PERDIDAS AL REINICIAR)
     Archivo: api/routes/licencias.py (~línea 73)
     Detalle: _licencias_activas es un dict en memoria Python.
              Al reiniciar el servidor, todas las activaciones se pierden.
     Fix: Crear modelo LicenseActivation en DB y persistir ahí.

H-03 | CSRF COOKIE ACCESIBLE POR JAVASCRIPT (httponly=False)
     Archivo: web/middleware/csrf.py (~línea 286)
     Detalle: Si existe XSS, el atacante puede leer el token CSRF del cookie.
     Fix: Cambiar httponly=True y proveer el token vía meta tag en HTML.

H-04 | CSP PERMITE 'unsafe-inline' EN SCRIPTS
     Archivo: web/middleware/security_headers.py (~línea 47)
     Detalle: script-src incluye 'unsafe-inline', anulando la protección CSP
              contra XSS completamente.
     Fix: Migrar scripts inline a archivos .js externos y usar nonces.

H-05 | CDN SCRIPTS SIN SRI (Subresource Integrity)
     Archivo: web/templates/base.html
     Detalle: Las librerías cargadas desde CDN (jsdelivr, etc.) no tienen
              hashes de integridad. Si el CDN se compromete, código malicioso
              se ejecuta en todos los gyms.
     Fix: Agregar atributo integrity="sha384-..." a todos los <script> CDN.

H-06 | NO HAY RATE LIMITING EN LOGIN
     Archivo: web/main_web.py
     Detalle: El endpoint de login no tiene rate limiting, permitiendo
              ataques de fuerza bruta contra credenciales de gyms.
     Fix: Aplicar rate_limit_dependency(attempts=5, window=900) al login.
          Implementar lockout temporal después de N intentos fallidos.

H-07 | NO HAY AUDIT LOG DE AUTENTICACIÓN
     Archivo: web/auth.py
     Detalle: No se loguean intentos de login (exitosos ni fallidos).
              Imposible detectar ataques de fuerza bruta o accesos no autorizados.
     Fix: logger.info("[AUTH] Login exitoso: %s") y
          logger.warning("[AUTH] Login fallido: %s")

H-08 | REMEMBER-ME TOKEN VÁLIDO 30 DÍAS
     Archivo: web/auth.py (~línea 304)
     Detalle: Access tokens con remember-me duran 30 días.
              Un token robado da acceso prolongado sin forma de revocar.
     Fix: Máximo 7 días para access tokens. Usar refresh token rotation.

H-09 | /metrics Y /health SIN AUTENTICACIÓN
     Archivo: api/app.py (líneas 136-200)
     Detalle: /metrics expone datos de rendimiento sin auth.
              Útil para reconocimiento por atacantes.
     Fix: Proteger /metrics con auth. Mantener /health público solo si
          es requerido por el load balancer.

H-10 | STRIPE SESSIONS EN MEMORIA (se pierden al reiniciar)
     Archivo: api/routes/pagos.py (~línea 69)
     Fix: Persistir session IDs en base de datos.

H-11 | DUAL DATABASE SIN COMPLETAR (Strangler Fig Pattern)
     Archivos: config/feature_flags.py, src/gestor_bd.py,
               src/repositories/cliente_repository.py
     Detalle: La migración de SQLite legacy a SQLAlchemy está en FASE 1
              (legacy como primary) y no hay mecanismo para avanzar fases.
              El módulo gestor_bd.py está marcado DEPRECATED pero sigue activo.
     Fix: Completar migración. Establecer timeline para deprecar legacy.

H-12 | PORT MISMATCH DOCKER vs CÓDIGO
     Detalle: Dockerfile EXPOSE ${PORT} (default 8000), pero hay referencias
              a puerto 8001 en config/settings.py.
     Fix: Estandarizar en 8000 para producción.

H-13 | CORS_ORIGINS INCLUYE LOCALHOST POR DEFAULT
     Archivo: config/settings.py (línea 50)
     Detalle: Los CORS origins por defecto incluyen localhost:8000 y
              localhost:8001, que no deberían estar en producción.
     Fix: En producción, CORS_ORIGINS debe ser solo el dominio real.

H-14 | SUBSCRIPTION MODEL SIN CAMPOS FISCALES (CFDI México)
     Archivo: web/database/models.py
     Detalle: Para vender en Guadalajara (México), se necesitan campos
              para RFC, razón social, régimen fiscal del gym para CFDI.
     Fix: Agregar campos fiscales al modelo de suscripción.

H-15 | RLS POLICIES NO ENFORZADAS A NIVEL APP
     Detalle: Las migraciones crean Row-Level Security policies en PostgreSQL,
              pero el código no establece app.current_tenant en la sesión.
              La aislación multi-tenant a nivel DB es inoperante.
     Fix: En get_db(), ejecutar: SET app.current_tenant = '{gym_id}'

H-16 | PAYMENT GATEWAY KEYS VACÍAS POR DEFAULT
     Archivo: config/settings.py (líneas 61-66)
     Detalle: STRIPE_SECRET_KEY y MERCADOPAGO_ACCESS_TOKEN defaultean a ""
              Si no se configuran, los handlers fallan silenciosamente.
     Fix: Validar al arrancar que las keys estén configuradas.

H-17 | REFRESH TOKEN CLEANUP INCOMPLETO
     Archivo: web/middleware/rate_limiter.py, web/auth.py
     Detalle: _cleanup_old_buckets() definida pero nunca ejecutada.
              Tokens expirados se acumulan indefinidamente.
     Fix: Programar tarea de limpieza periódica.

H-18 | FACTURA PDF CON DATOS PERSONALES
     Archivo: api/factura_pdf.py (~línea 161)
     Detalle: El footer del PDF de factura contiene datos personales del
              desarrollador en lugar del gym.
     Fix: Cargar datos del gym desde branding dinámico (DB).

13. RECOMENDACIONES ARQUITECTÓNICAS
================================================================================

1. MULTI-TENANT ARCHITECTURE:
   Para vender a múltiples gyms en Guadalajara, el modelo actual de
   "shared database with gym_id filtering" es correcto para el volumen
   esperado (decenas de gyms, no miles). NO necesitan schema-per-tenant
   ni database-per-tenant aún. Pero sí necesitan:
   - RLS activo en PostgreSQL como segunda capa de defensa
   - Branding dinámico cargado de DB
   - Aislamiento verificado con tests E2E

2. DEPLOYMENT:
   Railway es una buena opción para empezar. Para escalar:
   - Railway → AWS ECS o DigitalOcean App Platform cuando tengan 50+ gyms
   - PostgreSQL managed (Railway Postgres o Supabase) es suficiente para ahora
   - Redis solo cuando necesiten horizontal scaling (10+ pods)

3. MONETIZACIÓN:
   - Stripe es la opción correcta para pagos internacionales
   - MercadoPago es ESENCIAL para Guadalajara (domina el mercado mexicano)
   - Implementar AMBOS gateways antes de vender
   - Considerar OXXO Pay via Stripe para clientes sin tarjeta

4. COMPLIANCE MÉXICO:
   - CFDI (facturación electrónica) es OBLIGATORIO para cobrar a empresas
   - Necesitan RFC del gym para emitir facturas
   - Considerar integración con Facturapi o similar
   - Ley Federal de Protección de Datos Personales (LFPDPPP)
   - Aviso de privacidad OBLIGATORIO en la landing page

5. TESTING:
   1030 tests es excelente base. Agregar:
   - Tests de penetración (OWASP ZAP)
   - Load testing (k6 o locust)
   - Tests de migración de datos entre versiones
   - Tests de smoke en staging antes de promover a producción

6. PRIORIZACIÓN:
   El orden correcto es:
   1) Limpiar datos personales (URGENTE — riesgo legal)
   2) Fix secretos y seguridad crítica
   3) Completar pagos (Stripe real + MercadoPago)
   4) Multi-tenancy branding
   5) Deploy a staging
   6) Tests de penetración
   7) Deploy a producción
   8) Primer gym piloto
